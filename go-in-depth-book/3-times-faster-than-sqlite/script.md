# 一种比SQLite快3倍的文件结构

大家好。在AI时代，我们还需要学习诸如数据结构和算法之类的基础知识吗？我认为这些基础知识仍然非常重要。当然，AI可以帮我们解决一些工程上的问题，但架构方面还需要我们自己来设计。

例如Go语言的`map`类型，是一个哈希表，一个高度优化的哈希表。对于我们大多数人来说，除非你也要实现一个类似的哈希表，否则Go`map`的具体实现细节，例如桶分组、渐进扩容、溢出桶等，可能并不重要，但是哈希表的基本原理，例如单独链表法和开放寻址法，依然是我们需要深入理解的基础知识。

在我的新书《深入理解Go语言：规范、原理与实践》中，我使用这张图来介绍哈希表的单独链表法：

每个Key都被哈希函数映射到一个桶中，所有冲突的K-V对都被存储在一个链表中。这是我们《数据结构》课程中都学习过的。

其实，记住了这张图，对我们大多数程序员来说，就算真正理解了哈希表的工作原理。

有人说，学会了这个原理，除了可以写出一个demo级别的哈希表之外，还有什么用？

别急，我在写出了demo级别的哈希表之后，还参考上图设计了一种文件哈希表，或者说叫做“只读键值数据库”。

这种文件结构就是单独链表法的一个变体。为了能够在文件中快速定位各个桶，我在文件开头部分增加了一个桶索引区，存储每个桶的偏移量。

每个偏移量对应的桶中存储了一个数组结构，即个数+对应个数的项目。

每个项目中存储了一个key和一个value。由于value是变长的，所以先存储value的长度，再存储value的内容。

这样的数据结构在查询时，先计算出桶索引，然后根据偏移量seek到对应的桶位置，读取桶中的项目，找到对应的key，返回value。

我使用IEEE的OUI列表来测试这个文件结构的性能，发现在数万条记录的情况下，查询性能比SQLite快了3倍。

当然，这种比较是不公平的，因为SQLite是一个通用的关系型数据库，而这个文件结构是专门为键值查询设计的。但是，这个结果也说明了在特定场景下，设计一个定制化的解决方案可以大幅提高程序的性能。当然，这一切的前提是我们对数据结构和算法原理的理解。
