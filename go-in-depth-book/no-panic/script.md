# 还在使用panic？先保住工作要紧

大家好。

我们知道，在Go语言中，一旦一个函数调用了内建的panic函数，该函数的执行就被强行中止，如果又没有在上层捕获这个panic的话，整个程序就崩了。

那么，这个内建的panic函数的目的是什么？专门用来制造崩溃的？专门给你捣乱用的？那肯定不是啊。所谓不破不立。既然不是用来捣乱的，它到底有什么用？

在我的新书《深入理解Go语言 规范、原理与实践》中。有这么两个例子。

第一个，有这么一个函数，ConfigValue，它接收一个文件名configFile，然后从中读取出一个特定的配置值。

如果参数传入的文件无法打开，无论是文件不存在，或者没有权限，反正就是打不开，这里返回一个非nil的error，怎么办？有人说把这个error返回出去啊。对不起，很不幸，这个函数只返回一个值。那你改啊，改成返回两个值。对不起，改不了，为什么改不了？这个函数要作为回调函数使用，原型已经定死了，你返回两个值就编译不了了。把需要这个函数的地方也改了啊，对不起，影响太大，改不了。就算要改也要等咱们开个会来协调一下，也许下个版本才能该。那怎么办？

有人说，那文件打不开我就返回一个空字符串。万一空字符串代表一个有意义的配置值呢？这个错误的值被下游代码拿到，造成隐含的bug，在十万八千里之外出现一个不好复现的bug。那你就调试吧，能不能很快解决就看运气了。

为什么在出现错误时，返回一个没有良好定义的默认值是错误的编程实践呢？因为这种做法本质上是在说谎，明明在你的函数中发生了错误，你即没有暴露这个错误，也没能返回给上层，而是默默吞掉错误，并返回一个看似正常的值，这不是说谎是什么？

所以，在这种情况下，唯一可行的办法是使用panic抛出这个错误。这样做的好处在于及时暴露错误。例如在我们这个例子中，文件打不开意味着ConfigValue的调用者没有正确地给我们提供我们完成工作所必须的前置条件。就像老板让你去进货，结果给你的进货单是空的，你得暴露这个错误，去找老板要正确的进货单，而不是像AI一样预测一下，公司大概需要进什么货，然后自作主张。那你就麻烦大了。

有人说，你panic了，倒是好，程序崩溃了。对，就是要它崩溃，只有崩溃了，才能暴露错误，进而发现错误的根源，并解决错误。

你看一旦真的发生了panic，程序会输出Stack trace，在stack trace中不但给出了发生panic的行，同时也给出了是谁调用了我们，也就是谁传递了错误的参数。如果这参数又是从别的地方传递过来的，依然可以一路追踪过去。直到找到问题的根源。

我书中的另一个例子是这样的，一个生成下一个递增ID的函数NextID，它依赖一个全局maxID，每次为maxID加一，那么如果maxID已经达到uintptr最大值怎么办？我们当初选择uintptr类型，就是认为在我们有生之年不可能达到最大值。现在确实达到了最大值，要么是我们的选择错了，要么是这个maxID被谁手滑修改了。无论是那种，都意味着这个NextID函数无法继续执行了。此时也只能通过panic来及时暴露错误。

其实，曾经有人因为这个事和我抬杠（不，友好讨论），说要是我公司有规定，就是严格限制，绝对不让使用panic呢？哦。那确实是个问题，这年头找份工作不容易，你还是不要panic了，毕竟保住工作要紧。其实要放在几年前，我会理直气壮地告诉他，你还是换个公司吧。

感谢。
