# 软件工程没有用了吗？

大家好
问大家一个问题，在当下的2026年，软件工程还有用吗？有些人可能会说，什么软件工程，什么设计模式，都太教条了，早过时了。真的是这样吗？我们来看一个例子。

这里有一个appendEvil函数，它接收一个切片类型的参数s1，然后使用append函数为s1添加一些东西，然后返回结果切片。

main函数中有一个切片 s，为它赋初始值 It is OKAY，然后对s执行切片操作，得到一个小切片，它的内容为单词 is。接着调用刚才那个appendEvil函数在is的尾部添加一些evil的东西，把返回值保存在isEvil中。

最后输出结果isEvil和原始切片s。我们会惊奇地发现，明明我们只是对小切片is进行了append操作，结果原始切片s也被修改了。不得不说这个appendEvil真的很邪恶。

其实，这对于已经深刻理解切片的底层构造和append函数的实现原理的人来说，这些都非常好理解。

首先，切片之间是可以共享底层数组的，也就是说基于同一个底层数组生成的切片，它们的实际存储空间可能是共用的。

例如在这个例子中，小切片is是基于大切片s生成的，它俩就共享同样的底层数组array1，也就是说s和is都是array1的一个视图（view）。

然后是append，append函数在向一个切片中添加元素时，如果切片的容量足够，它会直接使用这个剩余容量。这个时候，如果这些剩余容量刚好就是另外一个切片的内容，那么一次append就会修改多个切片。

在我的书中，给出了两个解决方案。第一个就是在调用appendEvil时做好自身防护，不让appendEvil意外修改切片长度之后的内容。第二个是修改appendEvil本身，让它不那么evil，在append时不修改参数长度之后的存储空间。

虽然这些解决方案看起来好像是我们自己“发明”的，但仔细想想，第一种对参数做好防护，不就是不信任外部函数吗？通过防御性编程来抵御外部函数的副作用吗？第二种就更明显了，修改appendEvil，不让它把shit拉到自己地盘以外的地方，不就是我们当初怎么也背不会的“高内聚低耦合”吗？不是软件工程没有用了，是我们已经学会了，我们不能因为考上了大学，就鄙视自己的小学老师，也不能因为已经熟练使用软件工程的基本原则就说软件工程没有用了。

另外，还有一个结论也很明显。就算我们对软件工程了然于胸，如果没有对细节的把控，假如不理解go语言切片的底层结构，不懂append函数的底层原理，就算是使用AI写出了上述代码，那也只是空中楼阁，对自身的提升没有任何帮助。
