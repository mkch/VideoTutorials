# Go历史上寿命最短的特性

大家好，今天我们来聊聊我所见过的Go语言历史上寿命最短的特性。这个特性就是Go 1.26引入的对`go mod init`命令默认使用的Go工具链版本的更改。

Go1.26的 Release Notes明确提到`` `go mod init`现在在新建的模块中默认使用较低的Go版本 ``。

例如，Go 1.26 的候选版本会在 go.mod 文件中默认写入 go 1.24.0，而 Go 1.26 及其小版本则会写入 go 1.25.0。

Go官方给出的理由是为了提高新项目的兼容性和稳定性。

我为什么说它是历史上寿命最短的特性呢？因为这个特性是随着Go1.26于2026年2月10日发布，然而在接下来的3月5日，Go官方就发布了Go 1.26.1，并且在其中撤销了这个更改，还原了 `go mod init` 之前的行为。算下来，这个特性仅存活了不到一个月的时间。

虽然在Go1.26做出这个改动时，我也感到有些惊讶，但我作为一个Go语言的使用者，也无力改变这个事实。现在随着Go1.26.1的发布，这个更改已经被撤销，一切尘埃都落定了，我觉得是时候探究一下这个特性为什么会在这么短的时间内被撤销。

我查阅了 Go 1.26.1 里程碑下的 issue 列表，找到了一个起到决定性作用的 issue：<https://github.com/golang/go/issues/77653>。这个 issue 由社区贡献者 willfaught 发起，标题为“将 go mod init 默认的 go 指令改回 1.N”。

不得不说这位老哥非常聪明，他一上来就用一个实际的例子指出了这个更改导致的冲突点。他首先写了一个t.go文件，在其中使用了Go1.26引入的新特性 `new(42)`，然后使用go 1.26执行 `go run`，结果当然是成功的。接着他使用 `go mod init` 来为这个go文件创建了一个新模块，由于Go1.26的更改，go.mod文件中默认使用1.25工具链，于是接下来的 `go build` 就失败了。

同样的代码，同样的Go版本，`go run`成功了，但 `go build` 却失败了，不得不说这是一个非常巧妙的论据，可以证明Go1.26所做的修改并不自洽，并且让其他人不太好反驳。聪明人果然不一样，佩服！

当然，这位老哥并没有止步于此，他详细论述了这个更改不恰当的地方。其中核心原因就是，用户主动升级了Go版本，就说明他们想要使用新版的特性，“主动升级Go”这个行为本身就已经表明了用户的意图：如果我想使用旧的1.25的特性，那我为什么还要费劲升级到1.26呢？所以这个更改完全没有意义，反而会给用户带来困惑和不便。

当然，这个 issue 的讨论非常多，有100多条，感兴趣的朋友可以自行前往围观。不过最终的结果是，Go官方在3月5日发布的Go 1.26.1中撤销了这个更改，还原了 `go mod init` 之前的行为。

另外，这位老哥还提到一个非要有意思的事实：这个非常重要的更改在发布前的讨论热度却非常低，在emoji votes中只得到了6:6的平局，当然其中一个踩还是这位老哥自己点的。另外，这么重要的更改居然是在一个看起来像Proposal的议题中讨论的，而没有形成一个正式的proposal。

看到这一点，我最初是很震惊的，后来想想也觉得合理，毕竟Go的开发流程从来就不是社区投票决定的。我对这种“维护者主导”的开源治理模式一直保持中立态度，不知道大家这么看？欢迎在评论区留言讨论。
