# AI的上下文和记忆

我们知道，LLM语言大模型本身是没有上下文的。它只能根据输入的提示词进行推理和生成，而无法记住之前的对话内容或用户的偏好。但是我们常用的AI工具往往都有上下文和记忆功能。这些功能是如何实现的呢？

我最近在看一本书《图解大模型》，里面介绍了很多有意思的知识。我今天就给大家分享一下，AI的上下文和记忆是如何实现的。

## 上下文

由于常用的AI工具，例如Gemini、ChatGPT等，都已经实现了上下文功能，所以要展示没有上下文的AI，需要写代码直接调用API来实现。但是，今天我不想写代码，打算使用一些非常直观的手段为大家展示一下AI的上下文和记忆功能是如何实现的。

我在Gemini的Personal Context中关闭和记忆相关的功能。这样，在不同对话之间，Gemini就无法记住之前的对话内容了。我打算利用Gemini在不同对话之间的失忆来模拟没有上下文的AI工具。

首先我新开一个对话，告诉Gemini我叫小明。然后再问它一个问题，例如，1+1等于多少。没有问题。

当我们新开一个对话，再问它“我是谁？”的时候，它就不知道了。这就是在没有上下文时AI工具的表现。

那么上下文功能如何实现呢？我们找到之前的对话历史。把它复制出来，整理为这样一个提示词。然后追加我的新问题“我是谁？”。

把这个新的提示词输入到Gemini中。虽然这次依然是新开的对话，但是由于我们在提示词中嵌入了之前的对话历史，所以Gemini就能够知道我是谁了。

这就是AI工具上下文功能的实现原理。它通过在每次交互时自动加入之前的对话历史来实现上下文功能的。只不过这个过程被AI工具自动化了，我们看不到而已。

## 上下文的压缩

但是，我们知道LLM一次可处理的token数量，也就是提示词的长度，是有限的，当我们每次都把之前的完整对话历史加入到提示词中时，很快就会超过LLM的token限制了。为了应对这个问题，AI工具通常会对之前的对话历史进行压缩。

压缩的方法有很多种。最简单的就是窗口式压缩。也就是只保留最近N轮对话的历史，并抛弃最早的对话内容。

这么做的缺点显而易见。例如在我们刚才的例子中，现在已经存在3轮对话，如果限制窗口大小为2，那么最早的“我叫小明”这个信息就从上下文中丢失了。如果这个信息非常重要，那么随后的对话就会变得毫无意义了。

第二种做法是对话摘要。也就是对之前的对话历史进行总结和概括，提取出其中的关键信息，只把这些关键信息加入到后续的提示词中。做摘要，AI最擅长。我们把对话历史输入到AI中，并让它帮我们总结一下。我们使用总结过的对话历史来构建新的提示词。这样就可以在不超过token限制的情况下实现完整上下文功能了。

这么做也是有缺点的。所谓“总结”就是对之前的对话历史进行压缩和抽象，难免会丢失一些细节。这和压缩一个图片到jpeg格式一样。

其实我们人脑的记忆功能也大概如此：我举个现实的例子。前两天我老婆告诉我她在某个餐馆吃了一顿梅菜扣肉，真香。过了两天，她再和我谈起这个事情时，我只记得她那天吃了肉，真香。其中，菜的名字我已经忘记了。这就是对上下文进行总结和摘要的结果。

## 长期记忆

AI工具的长期记忆功能要更复杂一些。长期记忆会把我们和AI的对话内容转换为向量嵌入，并存放在向量数据库中。当我们再次和AI对话时，它从数据库中检索出与当前对话最相关的对话历史，并把它们加入到上下文中。这样就实现了长期记忆功能。关于向量嵌入和向量数据库的内容，就不在这里展开了。如果大家感兴趣，我也可以在后续的视频中为大家介绍一下。

一句话总结：所谓的AI上下文和长期记忆，都是通过把更多的相关信息加入到提示词中来实现的。
